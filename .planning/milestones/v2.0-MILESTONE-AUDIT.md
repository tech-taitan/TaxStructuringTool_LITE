## Cross-Phase Integration

**Connected:** 18 cross-phase exports properly imported and used
**Orphaned:** 2 exports never called (dead code, not blockers)
**Missing:** 1 connection (TP auto-flag at edge creation — compensated by validator Rule 8)

### Connected Exports (18)

Key wiring confirmed across all phases:
- entity-registry.ts → EntityPalette, MobilePalette, Canvas, RegistrationSection, TaxStatusSection, graph-validator
- jurisdiction.ts → Canvas, MobilePalette, PaletteItem, IdentitySection, CanvasLegend
- entity-schemas.ts → PropertiesPanel (getEntitySchema)
- cross-border.ts → ConnectionPropertiesPanel (COMMON_CURRENCIES, PAYMENT_TYPES)
- constants.ts → EntityNode, CanvasLegend (JURISDICTION_COLORS)
- relationship-schemas.ts → all 6 cross-border fields on base schema

### Orphaned Exports (2)

1. **`searchAllEntities`** in entity-registry.ts — palettes inline the same ENTITY_REGISTRY filter logic
2. **`isCrossBorder`** in cross-border.ts — detection inlined in RelationshipEdge.tsx and ConnectionPropertiesPanel.tsx

### Missing Connection (1)

**TP auto-flag at edge creation:** `getDefaultRelationshipData()` does not inspect source/target jurisdictions, so `transferPricingRelevant` defaults to `false` even on cross-border service connections. The UI label "Auto-flagged" is slightly misleading. However, graph-validator Rule 8 fires a warning correctly for this scenario, so the functional behavior is sound.
